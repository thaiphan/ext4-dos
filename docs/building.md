# Building & debugging ext4-dos

For people who want to hack on the code, build from source, or reproduce the test runs locally. If you're just here to use it, grab the `.EXE` files from a release and skip this doc.

---

## Build requirements

- **[OpenWatcom 2.0](https://github.com/open-watcom/open-watcom-v2/releases)** — builds the real-mode 16-bit DOS binaries. Install somewhere and set `WATCOM=` to that path.
- **GCC or clang** (`cc`) — builds the host-side test harness.
- **Python 3** — generates the test fixture disk images.
- **Linux `e2fsprogs`** (`mke2fs`, `e2fsck`, `debugfs`) — needed to create/validate ext4 images.
- **DOSBox-X** — for running the actual DOS tests. Our scripts pin to a specific upstream commit (see `scripts/setup-debugger.sh`).

OpenWatcom auto-detects the host via `uname -s` / `uname -m`:

| Host         | Bin dir           |
| ------------ | ----------------- |
| macOS arm64  | `$WATCOM/armo64/` |
| macOS x86_64 | `$WATCOM/bino64/` |
| Linux x86_64 | `$WATCOM/binl64/` |

Override with `WATCOM_BIN=` if your layout differs.

---

## Building

```sh
# DOS binaries (ext4.exe, ext4cli.exe, etc.) — requires Watcom
make dos

# Host-side test utilities
make host

# Everything
make
```

Output lands in `build/dos/` (DOS `.EXE` files) and `build/host/` (host test runners).

---

## Running the test suite

The test scripts automate booting DOS inside DOSBox-X, running a sequence of file operations, and checking `e2fsck -fn` on the resulting image.

```sh
# MS-DOS 4 smoke tests (writes, renames, wildcard DIR, etc.)
bash scripts/run-msdos4-test.sh

# FreeDOS smoke tests
bash scripts/run-freedos-test.sh
```

First time? You'll need the DOSBox-X build:

```sh
bash scripts/setup-debugger.sh    # ~10 min: clones + builds DOSBox-X with heavy-debug support
```

This pins to a specific DOSBox-X commit so the build is reproducible.

The fixture disk images that the tests run against are generated by the `scripts/mkfixture-*.py` scripts. The Makefile runs them automatically as needed.

CI builds for every push are available from the [Actions](../../actions) tab under the `dosix-binaries` artifact (requires GitHub auth).

## Debugging the TSR

Several of the tricky bugs we hit needed a real CPU debugger — interactive `BPM` watchpoints and single-stepping into MS-DOS 4 kernel code.

```sh
bash scripts/setup-debugger.sh    # one-off, builds DOSBox-X with --enable-heavy-debug
bash scripts/run-msdos4-debug.sh  # boots MS-DOS 4 in the interactive debugger
```

The heavy-debug build gives you `BPM` (break-on-memory-write) traps, which is how we traced things like the MS-DOS 4 IFS-header corruption bug. Details in [dos-internals.md](dos-internals.md).

## Project structure

```
src/               Core library (blockdev, ext4 fs layer, partition parsing)
tools/             DOS TSR (tsr.c) + all the DOS-side utility programs
build/dos/         Compiled DOS .EXE files
build/host/        Host-side test binaries
scripts/           Test automation + fixture generation
tests/             Fixture images and expected outputs
docs/              This and dos-internals.md
references/        Gitignored clones of lwext4, MS-DOS 4 source, FreeDOS kernel
```

The TSR entry point is `tools/tsr.c`. The ext4 implementation lives under `src/ext4/`. The DOS-side CLI tools (`tsr_*.c`) are thin shims over the same `INT 21h` / `INT 2Fh` interface that any DOS program would use.

## Adding a new file operation

1. Identify the `INT 2Fh AH=11h` subfunction number (check `docs/dos-internals.md` section 2).
2. Add a handler in `tools/tsr.c` following the existing pattern (dispatch by `AL`, read inputs from SDA/SFT, fill outputs, return `CF=0`).
3. Add a `tsr_*.c` utility to exercise it from DOS command line.
4. Add a test step to `scripts/run-msdos4-test.sh` and `scripts/run-freedos-test.sh`.
5. Check `e2fsck -fn` still passes.

See [dos-internals.md](dos-internals.md) for the full subfunction map and the SDA/SFT/CDS field layout you'll need.
